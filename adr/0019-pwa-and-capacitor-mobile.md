## Context

CrossEngin tenants are not always at desks. A pharmacist may scan barcodes on a phone in the storeroom; a field-service technician fills service tickets on a tablet at a customer site; a vaccination-clinic nurse records doses in a mobile-first workflow; a procurement officer reviews submissions on the way home. Mobile is not optional for several v1 verticals (pharmacy, field service, vaccination registry, construction site management).

The project's mobile constraints are explicit:

- **No native iOS / Android in v1.** PWA-first; Capacitor wrapper later.
- **Capacitor for mobile when needed** (per vision Year 3). Provides native plugins for BLE scanner, ESC/POS printer, NFC, camera.
- **Code reuse across desktop + mobile.** One codebase; viewport-aware layouts (per ADR-0018 renderer).
- **Offline-capable.** Pharmacy storerooms have spotty Wi-Fi. Field-service technicians lose signal. The app must keep working and sync when reconnected.
- **Fast updates.** PWA updates within minutes of a deploy without app-store review delays.

This ADR specifies the PWA architecture, offline-first patterns, the Capacitor strategy for Year 3+, and the native-plugin catalog.

## Decision

CrossEngin ships a **PWA-first mobile strategy** with a **Capacitor wrapper** for native distribution starting Year 3:

```
┌──────────────────────────────────────────────────────────────┐
│ Year 1 (PWA only)                                              │
│   - apps/web is installable on iOS/Android via standard PWA   │
│   - Service worker for offline + background sync              │
│   - Add-to-home-screen flow + push notifications              │
│   - No app-store presence                                      │
└──────────────────────────────────────────────────────────────┘
┌──────────────────────────────────────────────────────────────┐
│ Year 3+ (Capacitor wrapper)                                    │
│   - apps/mobile-shell wraps apps/web in Capacitor             │
│   - App Store (iOS) + Play Store (Android)                    │
│   - Native plugins: BLE scanner, ESC/POS, NFC, camera         │
│   - In-app updates via Capacitor Live Updates                 │
└──────────────────────────────────────────────────────────────┘
```

### Year-1 PWA scope

The `apps/web` Next.js app is **installable** as a PWA on iOS Safari 16.4+ and Android Chrome:

- **Web App Manifest** (`/manifest.webmanifest`) — name, short name, icons (multiple resolutions), theme color, display mode `standalone`, start URL.
- **Service worker** (`/sw.js`) generated by `@serwist/next` — handles offline, cache, background sync, push notifications.
- **HTTPS only.** Cloudflare-fronted TLS 1.3.
- **Add-to-home-screen prompt** — soft prompt after second visit; iOS users get the Safari Share-sheet path explained.
- **Push notifications** (Web Push API) — Year 2 once `notifications` are wired through ADR-0011 inbound webhooks.

### Service worker strategy

Three caching strategies, selected per request:

| Pattern | Strategy | Use |
|---|---|---|
| App shell (HTML, CSS, JS) | Stale-while-revalidate | Fast startup; new code applied on next nav |
| Static assets (images, fonts) | Cache-first | Versioned URLs; long max-age |
| API GET reads | Network-first w/ cache fallback | Fresh on connection; stale-but-usable offline |
| API POST/PUT/PATCH/DELETE writes | Network-only w/ offline queue | Queue writes for background sync |
| Auth endpoints | Network-only | Never cache |
| File downloads (R2 signed URLs) | Network-only | URLs are short-lived |

The service worker registers at `/sw.js` with scope `/`; controls the entire origin.

### Offline-first writes

The kernel exposes idempotent endpoints (per ADR-0011 idempotency). The offline queue:

1. Write request executed against local IndexedDB optimistically; UI reflects the result immediately.
2. Request added to `outbox` IndexedDB store with a generated idempotency key.
3. Service worker `sync` event fires when network is available; replays the queue against the kernel API.
4. On 2xx, mark queue item processed; on 4xx (permanent), surface error to user; on 5xx (transient), retry with backoff.
5. Real-time UI reconciliation: when the server response arrives, replace the optimistic record with the canonical version.

Conflict handling:

- **Last-write-wins** for simple updates by default.
- **Manual merge** for entities marked `traits: ["concurrencyConflictExposed"]` — the UI shows both versions and asks the user to choose.
- **Workflow transitions** are never retried optimistically — the server is authoritative; offline transitions queue and finalize on reconnect.

### IndexedDB layer

`packages/pwa` provides a typed IndexedDB wrapper:

- **`entityStore`** — per-entity-type IndexedDB store mirrors the tenant's hot data (recently viewed records, recently created records). Capped at 10K records per type; LRU eviction.
- **`fileStore`** — locally cached files for offline access (drug formularies, prescription scans the user just viewed).
- **`outbox`** — pending writes queue.
- **`auth`** — session token (encrypted with WebCrypto-derived key).
- **`searchIndex`** — minimal FTS index of cached entities for offline search.

IndexedDB is wrapped in `idb` (npm package). All access through the kernel client SDK; no raw IndexedDB code in apps.

### Sync engine

A background sync runs on:

- App foreground.
- Network reconnect.
- Push notification "data updated."
- Manual user-triggered refresh.

The sync engine reads the tenant's manifest + cached `updated_at` watermark + permission decisions; fetches deltas from `/api/v1/sync` (a CrossEngin-built endpoint that returns entities changed since the watermark, scoped by ABAC).

### Push notifications

- **Web Push API** for browser PWAs (Year 2 wire-up).
- **VAPID keys** managed in Supabase Vault.
- **Notification payload** scoped per-tenant: a notification's content respects the user's locale + RTL + permission-redacted fields.
- **Click-through** opens the relevant entity in the PWA.

### Performance budgets

- **App shell** (first-load JS + CSS) < 250 KB gzipped.
- **Time to Interactive** < 3 s on 4G.
- **Subsequent navigation** < 500 ms (cached assets).
- **Offline-to-online sync** < 2 s for typical queue size (< 50 items).
- **IndexedDB write** < 50 ms p95.

Lighthouse PWA score ≥ 90 enforced in CI (per ADR-0023).

### Year-3+ Capacitor wrapper

`apps/mobile-shell` is a thin Capacitor app that loads `apps/web` inside a native WebView:

- **iOS:** Capacitor + Xcode; deploys to App Store as "CrossEngin."
- **Android:** Capacitor + Gradle; deploys to Google Play.
- **Code sharing:** 100% of the web UI runs unchanged inside the wrapper; native plugins enhance.

#### Native plugins

| Plugin | Use |
|---|---|
| `@capacitor/camera` | Barcode/QR scanning, document photo capture |
| `@capacitor/filesystem` | Offline file storage with native APIs |
| `@capacitor/push-notifications` | iOS APNs + Android FCM push |
| `@capacitor-community/bluetooth-le` | BLE barcode scanner for pharmacies |
| `@capacitor-community/printer` | ESC/POS printer for receipts/labels |
| `@capacitor-community/nfc` | NFC ID-card reading for hospitals + ministries |
| `@capacitor/geolocation` | Site location for field service |
| `@capacitor/preferences` | Native key-value store for sensitive small data |
| `@capacitor-community/keep-awake` | Pharmacy-counter / kiosk mode |

Custom CrossEngin-authored plugins:

- **`@crossengin/capacitor-secure-storage`** — encrypted vault for offline-cached PHI when tenant policy permits.
- **`@crossengin/capacitor-mlllp`** — limited MLLP client for inbound HL7 on tablet-deployed clinics (rare; Phase 7+).

#### Capacitor live updates

Capacitor Live Updates (or `@capacitor/live-updates`) deploys JS/HTML/CSS updates without app-store review:

- App shell updates within minutes of `main` deploy.
- App-store reviews only required for native plugin changes (rare).
- Per-region update channels supported (UAE-resident tenants can be on different update cadence).

### Capacitor app-store presence

- **iOS App Store:** "CrossEngin" parent app with capability descriptions per family. Year 3+. Apple Developer Program enrollment.
- **Google Play:** same; managed by Google Play Console. Year 3+.
- **App-store privacy labels:** declared per data class CrossEngin handles. PHI, PII, location — full disclosure.

Family-specific apps (e.g., "CrossEngin Heal Vaccination Registry") may launch as separate App Store entries Year 4+ for tenant clarity. Architecture supports it; default is the unified app.

### Auth on mobile

PWA: session JWT in IndexedDB (encrypted). Refresh tokens via standard NextAuth refresh flow.

Capacitor: biometric unlock (`@capacitor-community/biometric-auth`) for app open after first login. Sessions stored in iOS Keychain / Android Keystore via `@capacitor/preferences` secure mode.

Per ADR-0008, MFA is required for compliance-bound roles. On mobile, WebAuthn supports platform authenticators (Face ID / fingerprint).

### Permission-aware mobile UX

The renderer (ADR-0018) handles permission decisions; on mobile the UI degrades the same way:

- Redacted fields show placeholders.
- Write-locked fields show locked icons.
- Unavailable workflow transitions hidden.

Mobile-specific UX patterns:

- **Bottom-sheet for filters and actions** instead of side-panels.
- **Pull-to-refresh** triggers sync.
- **Swipe-actions on list items** for common transitions (verify, dispense, cancel).
- **Bottom-tab navigation** for top-level sections.

### Offline + compliance constraints

Some compliance packs limit offline caching:

- **HIPAA-PHI offline caching:** tenant admin opts in; biometric unlock required on each app open; auto-purge after 24h inactivity.
- **21 CFR Part 11 e-signatures:** signature challenges require real-time server; offline signatures are not supported. Workflow transitions blocked when offline; queued reads allowed.
- **GxP audit:** offline operations still emit audit rows; the audit ID is provisional until reconnect (canonical ID issued on sync).

These constraints are enforced by `packages/auth` + `packages/compliance` regardless of mobile context.

## Alternatives considered

### Option A — Native iOS + Android apps from v1

Full Swift + Kotlin apps.

- **Pros:** Best mobile UX. Full platform integration.
- **Cons:** 3-4× the dev work (three codebases). Two app-store review cycles. Pre-revenue, can't sustain.
- **Why not:** PWA + Capacitor delivers 90% of value at 20% of the cost.

### Option B — React Native instead of Capacitor

Web for desktop, React Native for mobile, two codebases sharing some components.

- **Pros:** Better native rendering performance than WebView.
- **Cons:** Maintains two UI stacks. The renderer (ADR-0018) would need React Native variants. Year 3+ deliverable; not v1.
- **Why not:** Capacitor's "wrap the PWA" model preserves one codebase. Mobile-UX hot paths re-evaluated Year 3+; React Native is reconsidered then.

### Option C — Skip mobile entirely (desktop-only)

- **Pros:** Simplest.
- **Cons:** Pharmacy, field service, vaccination registry, construction site management all need mobile. Half the verticals.
- **Why not:** Mobile is required.

### Option D — Hybrid: web for desktop, Cordova / Ionic for mobile

- **Pros:** Mature hybrid stacks.
- **Cons:** Cordova is legacy; Ionic + Capacitor is the modern successor. Capacitor is the right choice in 2026.
- **Why not:** Capacitor wins on tooling and ecosystem.

### Option E — Flutter for mobile

- **Pros:** Excellent performance. Single codebase across platforms.
- **Cons:** Dart language; doesn't share TypeScript code with desktop. Renderer (ADR-0018) would need a parallel Flutter implementation. Massive duplication.
- **Why not:** PWA + Capacitor preserves the one-codebase win.

### Option F — Tauri 2.0 mobile

Tauri's mobile target (2.0 launched 2024).

- **Pros:** Smaller bundle than Capacitor. Rust-based plugins for security.
- **Cons:** Less mature; smaller plugin ecosystem in 2026.
- **Why not:** Capacitor's ecosystem (especially BLE, ESC/POS, NFC plugins) wins for our verticals.

## Consequences

### Positive

- **One codebase serves desktop + mobile.** Engineering velocity per feature is preserved across surfaces.
- **PWA fast updates.** Bug fixes ship to mobile users within hours of deploy, no app-store review.
- **Offline-first by construction.** Service worker + IndexedDB + sync engine handle network gaps without per-feature code.
- **Native plugins via Capacitor** cover the must-have hardware integrations (BLE, ESC/POS, NFC, camera) when needed.
- **Mobile-aware renderer** (ADR-0018) means no UI-fork for mobile.
- **App-store deferred to Year 3** matches revenue and demand.

### Negative

- **PWA on iOS has historical quirks.** iOS Safari adopted PWA features slower than Chrome. Mitigation: target iOS 16.4+ minimum (PWA push notifications shipped iOS 16.4); document degraded experience on older iOS.
- **Offline queue UX complexity.** Conflict resolution, optimistic UI errors, dropped operations — all are user-facing edge cases. Mitigation: explicit conflict-resolution UI; comprehensive error messaging.
- **Capacitor wrapper introduces a release cycle in Year 3.** App-store review delays for plugin changes. Mitigation: Live Updates cover the common case (JS-only changes).
- **Plugin maintenance burden.** Each Capacitor plugin is a dependency. Mitigation: prefer official `@capacitor/*` and `@capacitor-community/*`; minimize custom plugins.

### Neutral

- **PWA installability** is a soft adoption curve; not every user clicks "Add to Home Screen."
- **Mobile push notifications** require server-side VAPID + service worker; one-time setup.

### Reversibility

**Low cost** to evolve PWA caching strategies and offline-queue behavior.

**Moderate cost** to swap Capacitor for React Native or Flutter at Year 4+. Mobile-shell rewrite + plugin re-implementation; 3-4 months.

**High cost** to abandon mobile after first design partners adopt it.

## Implementation notes

- **Package locations:**
  - `packages/pwa` — service worker config, IndexedDB wrapper, sync engine, push-notification client.
  - `packages/capacitor-plugins` — custom Capacitor plugins (Year 3+).
  - `apps/web` — PWA manifest, service worker registration.
  - `apps/mobile-shell` — Capacitor wrapper (Year 3+).
- **Service worker generator:** `@serwist/next` Next.js plugin; configured via `next.config.mjs`.
- **VAPID keys:** generated once; public key shipped in PWA manifest; private key in Supabase Vault.
- **IndexedDB versioning:** schema version stored per database; upgrade migrations on version bump.
- **Sync endpoint:** `GET /api/v1/sync?since=<watermark>&entities=<list>` returns per-entity delta + new watermark. Permission-filtered.
- **Conflict-resolution UI:** rendered by `ui-renderers/conflict-merge` (added when needed).
- **Live Updates configuration:** Capacitor channel per environment (preview / staging / production) and per-region.
- **App-store assets:** App icons, screenshots, privacy labels, support URLs all maintained in `packages/capacitor-plugins/app-store-assets/`.
- **iOS / Android minimum versions:** iOS 16.4+ (PWA push); Android 10+. Documented in support matrix.
- **Testing:**
  - Service worker tests via Playwright with `serviceworker` API.
  - Offline simulation in CI: Playwright `setOffline(true)` + queue verification.
  - Capacitor test builds nightly (Year 3+) on GitHub Actions macOS runner for iOS, Ubuntu for Android.
  - Real-device smoke tests pre-Phase-5 launch.
