## Context

ADR-0259 (M4.12) shipped `crossengin gateway prune-idempotency
[--dry-run]` as a one-shot sweep of every row in
`meta.gateway_idempotency_records` where `expires_at < now()`. The
ADR explicitly listed three deferred scope concerns:

> Q1: Scope flags — `--operation-id <id>` / `--method <verb>` to
> prune only specific operation scopes (e.g. retire idempotency
> for a deprecated endpoint).
>
> Q2: `--limit <n>` cap on delete count (batch-delete pattern for
> very large prunes that risk lock contention).
>
> Q3: `--older-than <duration>` for force-prune records that
> haven't reached `expires_at` yet — risky for in-flight
> idempotency.

Operator workflows that need these:

- **Endpoint retirement.** When a deprecated operation (e.g.
  `legacy.orders.create`) is removed from the gateway routes,
  its idempotency records are still stuck in the table until the
  per-row TTL expires. With volume retention windows of 24-48h,
  that's a 1-2 day tail of "dead" rows. A scoped sweep cleans
  them in one command.

- **HTTP-verb-targeted prune.** Some operators run DELETE-only or
  PATCH-only idempotency policies (e.g. PATCH has 7d retention,
  POST has 24h). Pruning by method matches their operational
  model.

- **Lock-contention safety.** At very high write rates, a full
  table sweep can hold row-level locks long enough to block
  in-flight writes. A `--limit 10000` chunked-delete pattern run
  on a cron lets operators sweep in batches without blocking.

M4.13 closes Q1+Q2. Q3 (`--older-than`) is **explicitly NOT in
scope** — force-pruning before `expires_at` breaks the
idempotency contract for in-flight keys; if operators want that,
they take a deliberate per-incident decision via raw SQL with a
`--confirm` gate.

## Decision

Add three optional flags to the M4.12 CLI action: `--operation-id
<id>`, `--method <POST|PUT|PATCH|DELETE>`, `--limit <n>`. Extend
`PostgresIdempotencyStore.deleteExpired` and
`previewDeleteExpired` to accept a `PruneExpiredOptions` second
argument carrying the same three fields.

```ts
export interface PruneExpiredOptions {
  readonly operationId?: string;
  readonly method?: string;
  readonly limit?: number;
}

class PostgresIdempotencyStore {
  async deleteExpired(now: Date, options?: PruneExpiredOptions): Promise<number>;
  async previewDeleteExpired(now: Date, options?: PruneExpiredOptions): Promise<number>;
}
```

Backward-compat: callers passing only `now` get the byte-
identical M4.12 SQL string — verified by a test that asserts the
exact SQL when no options are passed.

**SQL shape.** Dynamic predicate construction via a shared
`buildExpiredScope(now, options)` helper that returns
`{whereTail, params}`:

```sql
-- baseline (no options)
DELETE FROM meta.gateway_idempotency_records WHERE expires_at < $1

-- with operationId
DELETE FROM meta.gateway_idempotency_records
WHERE expires_at < $1 AND operation_id = $2

-- with operationId + method
DELETE FROM meta.gateway_idempotency_records
WHERE expires_at < $1 AND operation_id = $2 AND method = $3

-- with limit (PG DELETE doesn't support LIMIT natively)
DELETE FROM meta.gateway_idempotency_records
WHERE record_id IN (
  SELECT record_id FROM meta.gateway_idempotency_records
  WHERE expires_at < $1
  LIMIT $2
)
```

The LIMIT case uses `record_id IN (SELECT ... LIMIT)` because PG
doesn't support `DELETE ... LIMIT` natively. `record_id` is the
unique TEXT key on the table (per ADR-0044 schema) — distinct
from the autogenerated UUID `id`, but operationally serves the
same role for this subquery (also indexed via the unique
constraint).

**Preview shape with LIMIT.** The preview SQL wraps the LIMIT in
a sub-SELECT so the count reflects what the DELETE would actually
remove (capped at LIMIT):

```sql
SELECT COUNT(*)::TEXT AS count FROM (
  SELECT 1 FROM meta.gateway_idempotency_records
  WHERE expires_at < $1 AND operation_id = $2
  LIMIT $3
) sub
```

This is essential for the dry-run → live workflow — operators
should see the same number twice if nothing else changes.

**CLI validation.**

| Flag | Rule | Failure exit |
|------|------|--------------|
| `--operation-id` | Any non-empty string | _none_ (operator's responsibility — typos surface as 0 deleted) |
| `--method` | Must be `POST` / `PUT` / `PATCH` / `DELETE` (the schema CHECK values) | exit 2 |
| `--limit` | Positive integer (>= 1) | exit 2 |

Validation fires **before** any PG connection / store
instantiation — invalid flags exit 2 with no side effects.
`--method GET` and `--limit 0` / `--limit 1.5` / `--limit abc`
all return exit 2 with a clear error message; no PG queries.

**Output.** Both human and JSON formats echo the scope:

```
$ crossengin gateway prune-idempotency --operation-id tenants.create --method POST --limit 50
deleted 17 expired idempotency record(s) (as of 2026-05-26T12:00:00.000Z; scope: operationId=tenants.create, method=POST, limit=50)

$ crossengin gateway prune-idempotency --operation-id tenants.create --format json
{
  "action": "gateway.prune-idempotency",
  "dryRun": false,
  "asOf": "2026-05-26T12:00:00.000Z",
  "deletedCount": 17,
  "operationId": "tenants.create",
  "method": null,
  "limit": null
}
```

JSON always emits all three scope fields (`null` when unset) so
consumers don't need defensive existence checks. Human format
appends a `; scope: ...` suffix only when at least one filter is
set — clean output for the common no-filter case.

## Alternatives considered

- **Validate `--operation-id` against a registry of known
  operationIds.**
  - **Why not:** the gateway runtime has the registry (M4.7), but
    the CLI doesn't currently consult it for this action. Adding
    that lookup would require either embedding the registry in
    architect-cli or making the action PG-aware about routes.
    Typos surface naturally as 0-row prunes (loud at the
    operator's terminal); registry validation is a deferred
    convenience.

- **Add `--tenant-id <uuid>` for tenant-scoped pruning.**
  - **Why not:** the table has RLS + tenant_id NOT NULL, and the
    operational model is platform-wide audit. Per-tenant pruning
    is a sensible future Q (e.g. for tenant offboarding) but
    crosses into territory better handled via the retention
    substrate's per-tenant policies (ADR-0155). Defer.

- **Use `DELETE ... RETURNING` to surface the deleted rows.**
  - **Why not:** at LIMIT 10000+ the return payload is large and
    not used. Operators wanting forensic record of what was
    deleted use `--dry-run` first and capture the count, then
    run live.

- **Apply `--limit` semantics as "delete N oldest expired".**
  - **Why not:** the table doesn't have a natural ordering beyond
    `expires_at`. The `id IN (SELECT ... LIMIT)` shape lets PG
    pick any N rows — fine for prune semantics (all are
    eligible). Adding `ORDER BY expires_at` would force a sort
    on a partial index that doesn't exist today.

- **Implement `--older-than <duration>` (ADR-0259 Q3).**
  - **Why not:** force-prune before `expires_at` breaks the
    idempotency contract for keys still in their TTL window.
    Operators wanting that behavior take it deliberately via
    raw SQL with a `--confirm` gate.

- **Multi-value `--operation-id` (`--operation-id a,b,c` or
  repeated flag).**
  - **Why not:** simpler to chain via shell (`for op in a b c;
    do ... --operation-id $op; done`). Multi-value is a clear
    future Q (deferred) but doesn't gate the M4.13 value
    proposition.

- **Skip the limit and let the cron schedule act as the cap.**
  - **Why not:** without `--limit`, a single delete can hold
    locks for minutes. The cron cadence doesn't help mid-sweep.
    Explicit `--limit` is the operator-facing safety knob.

## Consequences

- **Positive:** scoped prunes are now first-class. Endpoint
  retirement + verb-specific policies + chunked-delete safety all
  achievable in one command.
- **Positive:** the adapter contract widens additively
  (`options?: PruneExpiredOptions`) — no breaking change. Existing
  M4.12 callers continue to work; the existing M4.12 SQL string
  is byte-identical when no options are passed.
- **Positive:** validation fires at the CLI boundary before PG
  resolution; operator mistakes (`--method GET`, `--limit 0`) are
  caught loudly with no side effects.
- **Neutral:** JSON envelope grows by 3 fields (operationId,
  method, limit) — always present, `null` when unset.
  Consumers reading the M4.12 envelope continue to work
  (the new fields are pure additions).
- **Neutral:** the LIMIT subquery shape adds modest plan
  complexity. The `record_id` unique index makes the inner
  SELECT + outer DELETE both index-only at modest sizes; at very
  high counts, time-based partitioning (ADR-0259 Q5) is the
  better answer.
- **Reversibility:** trivial — revert the `PruneExpiredOptions`
  parameter (drop to `now`-only), the three CLI flags, and the
  scope-suffix renderer. The substrate stays usable in M4.12
  form.

## Implementation notes

- `buildExpiredScope` is a private adapter helper; the bind
  order is always `[now, operationId?, method?]` with `limit?`
  appended last when the LIMIT subquery is used. Tests assert
  this verbatim.
- The CLI uses the existing `getStringFlag` helper for all three
  flags. `--operation-id` is unvalidated (any non-empty string);
  `--method` matches against a 4-value const tuple
  `IDEMPOTENCY_METHODS`; `--limit` parses via `Number(...)` with
  `Number.isInteger && > 0` gate.
- The `record_id` column (TEXT, unique constraint, format
  `idem_[A-Za-z0-9_-]{8,64}`) is the key used by the
  `IN (SELECT ... LIMIT)` subquery — distinct from the
  autogenerated UUID `id` column. Operationally either would
  work; `record_id` matches the existing INSERT/SELECT/UPDATE
  bindings throughout the adapter.
- 8 new adapter tests under "prune scope options (M4.13)":
  byte-identical SQL with no options; operationId-only; method-
  only; both combined; limit-only (subquery shape); all three;
  preview with limit wraps in sub-SELECT; preview without limit
  but with method emits simpler SELECT.
- 8 new CLI tests under "prune-idempotency scope flags
  (M4.13)": --operation-id threads + appears in output;
  --method threads; invalid method exits 2 with no PG call;
  --limit threads with subquery SQL; invalid --limit (0,
  non-numeric, float) all exit 2 with no PG call; all three
  combine with correct bind order; JSON envelope echoes all
  three (null when unset); dry-run JSON includes scope +
  wouldDeleteCount.
- CLI smoke-tested:
  `crossengin gateway prune-idempotency --method GET` → exit 2;
  `--limit -5` → exit 2; valid flags + missing PG env → exit 1
  with PG-env error (validation fires before PG).
- workspace test count 9,491 → **9,507**. api-gateway-pg 77 → 85
  (+8). architect-cli 1,233 → 1,241 (+8).
